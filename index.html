Web Services in SAP ABAP - Overview

  • Integration is essential in modern enterprise landscapes
  • SAP systems need to communicate with:
    • Other SAP systems
    • Non-SAP systems
    • Cloud applications
    • Mobile applications
  • Web services provide standardized communication protocols
  • SAP ABAP supports three main web service technologies:
    • SOAP (Simple Object Access Protocol)
    • REST (Representational State Transfer)
    • OData (Open Data Protocol)

API Protocols Evolution in SAP ABAP

Each protocol serves distinct purposes, with varying levels of complexity, flexibility, and alignment with SAP's evolving architecture. Selection should be based on technical requirements, integration targets, and long-term strategic direction.

SOAP Web Services

What is SOAP?

  • Simple Object Access Protocol
  • XML-based messaging protocol for exchanging structured information
  • Uses Web Service Definition Language (WSDL) to describe the service
  • Typically runs over HTTP/HTTPS, but can use other transport protocols
  • Follows strict standards and formats

Key Characteristics

  • Strongly typed interface definition
  • Protocol independent (can run over HTTP, SMTP, etc.)
  • Built-in error handling (SOAP faults)
  • WS-* standards support (WS-Security, WS-ReliableMessaging, etc.)
  • Stateless by design, but can maintain state through headers
  • Strong security: Supports WS-Security for encryption, authentication, and compliance with enterprise security policies
  • ACID transactions: Ensures reliable, stateful operations for mission-critical processes like financial transactions
  • Formal contracts: WSDL files specify input/output parameters, enabling strict validation

SOAP in SAP ABAP

Implementation Options

  1. ABAP Proxy Framework
    • Generated from WSDL or ABAP class
    • Client and server proxy generation
    • Transaction: SPROXY
  2. Web Service Framework
    • Transaction: SOAMANAGER
    • Wizard-based approach
    • Supports both client and server scenarios

Implementation Workflow

  1. Service Creation: Use transaction code SOAMANAGER to generate SOAP web services from ABAP function modules
  2. WSDL Generation: Expose service metadata via WSDL for client applications
  3. Testing: Validate SOAP requests/responses using tools like SOAPUI or ABAP's built-in test utilities

RESTful APIs

What is REST?

  • REpresentational State Transfer
  • Architectural style for distributed systems
  • Uses standard HTTP methods (GET, POST, PUT, DELETE)
  • Typically uses JSON or XML for data format
  • Relies on HTTP status codes for error handling

Key Characteristics

  • Resource-based (URLs represent resources)
  • Stateless communication
  • Uniform interface (HTTP methods)
  • Cacheable responses
  • Client-server separation
  • Layered system architecture
  • Lightweight: JSON payloads reduce parsing overhead compared to SOAP's XML
  • Statelessness: Each request contains all necessary context, simplifying horizontal scaling
  • Caching: HTTP caching mechanisms improve performance for read-heavy operations

REST in SAP ABAP

Implementation Options

  1. SAP Gateway
    • Transaction: SEGW
    • Model-driven development
    • Can build custom REST services
  2. ABAP RESTful Programming Model
    • Introduced in SAP NetWeaver 7.50+
    • Path to SAP S/4HANA extensions
    • Uses CDS views and behavior definitions
  3. ICF Services
    • Manual implementation
    • More control, but more complex
    • Transaction: SICF

Implementation Workflow

  1. HTTP Communication: Use class CL_HTTP_CLIENT or CL_REST_HTTP_CLIENT to call REST APIs
  2. JSON Handling: Parse responses with /UI2/CL_JSON or transform data using XSLT
  3. Security: Implement OAuth 2.0 or API keys via HTTP headers

Example: Creating a RESTful Service in ABAP

CLASS zcl_rest_service DEFINITION
  PUBLIC
  CREATE PUBLIC
  INHERITING FROM cl_rest_http_handler.

  PUBLIC SECTION.
    METHODS: if_rest_resource~get REDEFINITION,
             if_rest_resource~post REDEFINITION.
ENDCLASS.

CLASS zcl_rest_service IMPLEMENTATION.
  METHOD if_rest_resource~get.
    DATA: lv_id TYPE string,
          ls_customer TYPE bapicustomer.
          
    " Get ID from URI parameters
    lv_id = mo_request->get_uri_attribute( 'id' ).
    
    " Business logic to fetch customer data
    
    " Set response
    mo_response->set_status( 200 ).
    mo_response->set_content_type( 'application/json' ).
    mo_response->set_body( '{"id":"' && lv_id && '","name":"Customer Name"}' ).
  ENDMETHOD.

  METHOD if_rest_resource~post.
    " Implementation for POST method
  ENDMETHOD.
ENDCLASS.

OData Services

What is OData?

  • Open Data Protocol
  • REST-based protocol for querying and updating data
  • Originally developed by Microsoft, now OASIS standard
  • SAP is a major contributor and adopter
  • Uses entity-relationship model (similar to database tables)

Key Characteristics

  • Standardized CRUD operations
  • Rich query capabilities ($filter, $select, $expand, etc.)
  • Metadata description (EDMX)
  • Batching of multiple operations
  • Pagination support
  • Relationships between entities
  • Query options: $filter, $expand, and $select for client-driven data shaping
  • Metadata: $metadata endpoint describes entity models and relationships
  • Batch processing: Combine multiple operations in a single HTTP request

OData in SAP ABAP

Implementation Options

  1. SAP Gateway
    • Transaction: SEGW
    • Model-driven development
    • Generates required artifacts
  2. ABAP RESTful Programming Model
    • CDS Views with OData annotations
    • Behavior definitions
    • Business objects expose as OData
  3. SAP Gateway Service Builder
    • Transaction: /IWFND/MAINT_SERVICE
    • Service registration and activation

Implementation Workflow

  1. Service Development: Use transaction SEGW to define OData entities and map them to CDS views or ABAP logic
  2. Business Logic: Implement methods like GET_ENTITYSET in generated DPC_EXT classes
  3. Testing: Validate services using the SAP Gateway Client (/IWFND/GW_CLIENT)

Example: Creating an OData Service with CDS Views

@AbapCatalog.sqlViewName: 'ZCDS_CUSTOMER'
@AbapCatalog.compiler.compareFilter: true
@AccessControl.authorizationCheck: #CHECK
@EndUserText.label: 'Customer Data'
@OData.publish: true
define view ZCustomer as select from kna1 {
  key kunnr as CustomerID,
  name1 as CustomerName,
  land1 as Country,
  ort01 as City
}

With behavior definition:

managed implementation in class zcl_customer unique;

define behavior for ZCustomer alias Customer
persistent table kna1
lock master
authorization master
{
  create;
  update;
  delete;
}

OData Example Query

Example OData Query

/odata/Categories?$expand=Products($orderby=Price desc;$top=5)

This query:

  • Retrieves all Categories
  • Expands the related Products for each category
  • Orders the products by Price in descending order
  • Limits to the top 5 products per category

More Query Examples

# Filter for customers in a specific country
/odata/Customers?$filter=Country eq 'US'

# Select only specific fields
/odata/Products?$select=ID,Name,Price

# Pagination with skip and top
/odata/Orders?$skip=20&$top=10

# Complex filter with multiple conditions
/odata/Invoices?$filter=Amount gt 1000 and Status eq 'Open'

# Expand multiple navigation properties
/odata/SalesOrders?$expand=Customer,Items,Partner

Use Case

Fiori applications displaying aggregated supplier data with nested purchase orders, dashboards requiring complex data querying, and integration with SAP UI5 applications.

Detailed Comparison

Feature SOAP REST OData
Data Format XML only JSON, XML, others JSON, XML (Atom)
Service Description WSDL No standard (OpenAPI) EDMX metadata
Transport Protocol Any (typically HTTP) HTTP/HTTPS HTTP/HTTPS
Operations Custom methods HTTP methods HTTP methods + query options
Complexity High Low Medium
Performance Lower (XML overhead) Higher Medium
Built-in Features Many (security, transactions) Minimal Moderate (query options)
Learning Curve Steep Shallow Moderate
Security WS-Security, SAML OAuth 2.0, JWT Inherits REST security
Standards WSDL, WS-* specs HTTP methods, URI conventions OData v4/v2, REST principles

SOAP vs. REST vs. OData in SAP

SOAP Advantages in SAP

  • Well-established in SAP environments
  • Strong tooling support in older SAP systems
  • Formal contract through WSDL
  • Comprehensive WS-* standards support
  • Better for complex transactions
  • Legacy compatibility but declining SAP focus

REST Advantages in SAP

  • Lightweight and simple to implement
  • Better performance
  • Easier to consume by modern clients
  • More flexible data formats
  • Simpler debugging

OData Advantages in SAP

  • SAP's preferred standard for new developments
  • Rich querying capabilities out-of-the-box
  • Standard approach for SAP Fiori applications
  • Balances standardization and simplicity
  • First-class support in SAP Gateway
  • Central to ABAP RAP and S/4HANA Cloud

  lr_server->set_controller_object( me ).
  lr_server->receive_request( request ).
  lr_server->dispatch_method( ).
  lr_server->send_response( response ).

* Example business method
METHOD get_customer_details.
  SELECT SINGLE * FROM kna1 
    INTO CORRESPONDING FIELDS OF es_customer

  ENDIF.

  " Get filter conditions from request

When to Use Each Technology in SAP

Choose SOAP When:

  • Working with legacy SAP systems (pre-NetWeaver 7.3)
  • Complex operations with multiple steps
  • Formal contracts are required
  • Advanced security requirements exist
  • Reliable messaging is needed
  • Operation-centric rather than resource-centric
  • B2B integrations requiring ACID compliance

Choose REST When:

  • Simple, resource-oriented services
  • Performance is critical
  • Integrating with modern web/mobile applications
  • Flexibility is more important than standardization
  • Implementing custom lightweight APIs

Choose OData When:

  • Building SAP Fiori applications
  • Need for standardized query capabilities
  • Working with SAP Gateway
  • Complex data models with relationships
  • Avoiding custom implementation of CRUD operations
  • Following SAP's strategic direction

Best Practices for Web Services in SAP ABAP

General Best Practices

  • Document your services thoroughly
  • Implement proper error handling
  • Consider security from the beginning
  • Test performance under load
  • Version your APIs

SOAP Best Practices

  • Keep message sizes small
  • Limit use of SOAP headers
  • Implement proper exception handling
  • Consider WS-Security for sensitive data

REST/OData Best Practices

  • Use meaningful resource URIs
  • Implement proper HTTP status codes
  • Consider ETags for concurrency
  • Implement pagination for large results
  • Use proper content negotiation
  • Leverage CDS Views with OData to optimize data retrieval via AMDP or database hints

Recommendations and Future Roadmap

For New Developments:

  • First choice: OData with ABAP RESTful Programming Model
    • Strategic direction for SAP
    • Best integration with Fiori
    • Most modernized approach
  • Second choice: Custom REST APIs
    • When OData is too heavyweight
    • For specialized integration needs
    • When more control is needed
  • Last choice: SOAP
    • Only when specific SOAP features are required
    • For backward compatibility

Strategic Recommendations:

  • Adopt OData v4 for new projects due to SAP's roadmap emphasis
  • Migrate Legacy SOAP to REST/OData using tools like DreamFactory or SAP Gateway
  • Leverage CDS Views with OData to optimize data retrieval via AMDP or database hints
  • Prioritize Security: Use WS-Security for SOAP and OAuth 2.0 for REST/OData

For Existing Systems:

  • Consider your consumers' capabilities
  • Balance modernization with practicality
  • Possibly implement facades to legacy services

Summary

  • SOAP:
    • XML-based, well-defined standards
    • More complex, higher overhead
    • Still relevant for complex enterprise scenarios and legacy systems
  • REST:
    • Lightweight, resource-oriented
    • Simple to implement and consume
    • Great for straightforward integration scenarios
  • OData:
    • Standardized REST-based protocol
    • Rich querying capabilities
    • SAP's strategic choice for modern applications

Conclusion

SOAP, REST, and OData each occupy distinct niches in SAP ABAP ecosystems. While SOAP remains relevant for transactional integrations, REST and OData dominate modern architectures due to their flexibility and alignment with cloud-first strategies. Developers should prioritize OData v4 for future-proofing SAP integrations while maintaining SOAP for legacy compliance. By understanding these protocols' strengths, teams can design scalable, secure, and performant interfaces across hybrid SAP landscapes.
